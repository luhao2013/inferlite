# M2 — KV Cache

> **一句话目标**：把 M1 的"每步 full forward"改成"一次 prefill + N 次轻量 decode"，decode 阶段单步 Attention 复杂度从 O(T²) 降至 O(T)，同时支持 MPS/CUDA + bf16 推理。
>
> **完成日期**：2026-06-29
> **设计文档（含完整 ADR）**：[m2-kv-cache-design.md](m2-kv-cache-design.md)
> **实测结果归档**：[bench/results/2026-06-29-m2-kv-cache-mps-bf16.md](../../bench/results/2026-06-29-m2-kv-cache-mps-bf16.md)

## 符号说明

| 符号 | 含义 | Qwen3-0.6B 值 |
|------|------|--------------|
| B | batch size | 1（推理通常 B=1） |
| T_p | prompt 长度（prefill 阶段 token 数） | 100～1024 |
| N | 最大生成 token 数 | 100～512 |
| T | 当前序列总长 = T_p + 已生成 token 数 | ≤ L_max |
| L_max | max_seq_len，KV Cache 最大容量 | 1024（默认） |
| L | num_hidden_layers | 28 |
| H | hidden_size | 1024 |
| H_q | num_attention_heads | 16 |
| H_kv | num_key_value_heads（GQA < MHA） | 8 |
| D | head_dim = H / H_q | 64 |

## 1. 背景：M1 的浪费在哪里

### 1.1 自回归推理的两个阶段

LLM 生成文本的过程分为两个截然不同的阶段：

| 阶段 | 输入规模 | Q、K、V 形状（单层，GQA） | Attention 计算量 | 可缓存？ |
|------|---------|------------------------|----------------|---------|
| **Prefill** | T_p 个 token 并行 | Q: [B, H_q, T_p, D]，K/V: [B, H_kv, T_p, D] | O(T_p²)，T_p×T_p 矩阵乘 | 否——token 间两两计算，无法跳过 |
| **Decode**（每步） | 1 个 token | Q: [B, H_q, 1, D]，K/V: [B, H_kv, T, D]（T 含历史） | O(T)，1×T 向量乘 | **是**——历史 K/V 不受新 token 影响 |

KV Cache **只优化 decode 阶段**，不影响 prefill。

### 1.2 M1 的浪费

M1 无 KV Cache，每个 decode 步都把完整历史序列过一遍 Attention：

```
decode 第 k 步（k = 1..N）：
  输入序列长度 = T_p + k
  每层 Attention：Q[B, H_q, T_p+k, D] @ K^T[B, H_kv, T_p+k, D]，计算量 O((T_p+k)²)
```

N 步总 Attention 计算量：

```
M1 总量 ≈ Σ_{k=1}^{N} (T_p + k)² ≈ O(N · T²)    其中 T = T_p + N
```

**浪费的根源**：decode 第 k 步中，前 T_p+k-1 个 token 的 K/V 向量与第 k-1 步计算结果**完全相同**（causal mask 保证位置 t 的 K/V 只依赖 0..t，不受后续 token 影响），但 M1 每步都从头重算。

### 1.3 KV Cache 的效果

缓存 K/V 后，decode 第 k 步变为：

```
1. 只计算当前 1 个 token 的 Q：shape [B, H_q, 1, D]
2. 读取 cache 中前 T_p+k-1 个 K/V：shape [B, H_kv, T_p+k-1, D]
3. Attention：Q[1×(T_p+k-1)] 向量乘，计算量 O(T)
4. 把当前 token 的 K/V 追加写入 cache 第 T_p+k 位置
```

N 步总计算量：

```
M2 总量 = O(N · T)    比 M1 的 O(N · T²) 少了 T 倍
```

**为什么只 cache K 和 V，不 cache Q？**

Q（Query）代表"当前 token 想查什么"，每步生成的新 token 不同，Q 必须重新计算。
K/V 代表"历史 token 的记忆"，由 causal mask 保证位置 t 的 K/V 仅依赖 0..t 的 token，在 decode 过程中**永远不变**，因此可以且只有 K/V 可以缓存。

## 2. 显存占用与 L_max 选择

### 2.1 显存计算公式

KV Cache 为每层、每个位置预留 K/V 存储空间：

```
显存(bytes) = 2 × B × L × L_max × H_kv × D × dtype_bytes
              ↑K+V  ↑批   ↑层数  ↑最大长度  ↑KV头数 ↑头维度
```

**Qwen3-0.6B 参数代入**：

| 参数 | 符号 | 值 | 系数说明 |
|------|------|---|---------|
| K+V 两份 | — | 2 | K 和 V 各一份 |
| batch size | B | 1 | 单序列推理 |
| 层数 | L | 28 | 每层独立缓存 |
| 最大序列长度 | L_max | 1024 | **最大影响因子**，线性 |
| KV head 数 | H_kv | 8 | GQA：H_q=16 共享为 H_kv=8 |
| 每头维度 | D | 64 | head_dim |
| dtype | — | bf16 = 2 bytes | fp32 翻倍 |

```
Qwen3-0.6B，B=1，L_max=1024，bf16：
  2 × 1 × 28 × 1024 × 8 × 64 × 2 = 58 MB

若 L_max=4096：  58 × 4 = 233 MB
若用 MHA（H_kv=16）：58 × 2 = 117 MB   ← GQA 的节省体现于此
```

> **注**：Qwen3-0.6B 的 `head_dim` 实为 64（H=1024，H_q=16，D=H/H_q=64），不要与文档里偶尔出现的 128 混淆（那是 Qwen2 的值）。

### 2.2 GQA 对显存的影响

GQA（Grouped Query Attention）：H_kv < H_q，多个 Q head 共享同一组 K/V head。Qwen3-0.6B 中 H_q=16、H_kv=8，每 2 个 Q head 共享 1 对 KV。

KV Cache 大小与 H_kv 成正比，**GQA 直接把 cache 显存减半**（相比 MHA H_kv=H_q=16）。这不是巧合——GQA 的设计动机之一就是在不大幅损失精度的前提下缩小 KV Cache。

### 2.3 L_max（max_seq_len）如何确定

L_max 是静态预分配方案的关键参数，**必须在 generate 之前由调用方指定**：

```
T_p + N ≤ L_max
   ↑          ↑
prompt长度  最大生成数   需满足此约束，否则 cache 越界报 IndexError
```

| 做法 | 适合场景 | 缺点 |
|------|---------|------|
| 固定 L_max（如 1024） | 大多数对话场景，默认推荐 | 超长 prompt 会报错 |
| L_max = T_p + max_new_tokens | 完全精确，零浪费 | 每次 generate 需额外传参 |
| L_max = model.config.max_position_embeddings | 模型支持的最大值 | 显存最大（Qwen3-0.6B 为 40960） |

M2 策略：`from_config` 工厂方法接收 `max_seq_len` 参数，默认值 1024，可由 `--max-seq-len` CLI 参数覆盖。若 T > L_max 触发 IndexError，明确报错优于静默截断。

## 3. 数据结构设计

### 3.1 LayerKVCache + KVCache

```python
# inferlite/model/kv_cache.py

@dataclass
class LayerKVCache:
    """单层的 K/V 缓存。每层独立，调试时可单独检查。"""
    k: torch.Tensor  # [B, n_kv_heads, max_seq_len, head_dim]
    v: torch.Tensor  # [B, n_kv_heads, max_seq_len, head_dim]

class KVCache:
    def __init__(self, layers: list[LayerKVCache]) -> None:
        self.layers = layers
        self.cur_len: int = 0  # 已写入多少个 token，唯一事实源

    @classmethod
    def from_config(cls, config, batch_size, max_seq_len, dtype, device) -> KVCache:
        """按模型配置预分配所有层的 K/V tensor。"""
        layers = [
            LayerKVCache(
                k=torch.zeros(batch_size, config.num_key_value_heads,
                              max_seq_len, config.head_dim, dtype=dtype, device=device),
                v=torch.zeros(batch_size, config.num_key_value_heads,
                              max_seq_len, config.head_dim, dtype=dtype, device=device),
            )
            for _ in range(config.num_hidden_layers)
        ]
        return cls(layers)

    def reset(self) -> None:
        self.cur_len = 0  # tensor 不清零，下次 prefill 会覆盖写入
```

**为什么静态预分配？**
- decode 阶段零内存分配（切片写入不触发 `torch.cat`，避免每步申请新内存）
- 内存地址固定（潜在支持 cudagraph/torch.compile）
- 代价：需提前指定 L_max，超出时报错而非动态扩容

**为什么不清零 tensor？**
- `reset()` 只清 `cur_len = 0`，旧 tensor 数据保留
- 下次 prefill 会覆盖写入 [0, T_p) 范围，旧数据不影响正确性
- 避免 `fill_(0)` 的额外开销（58 MB × fill = 非必要 I/O）

### 3.2 内存布局

```
预分配 tensor：shape [B=1, H_kv=8, L_max=1024, D=64]

位置索引：  0    1    2   ...  T_p-1  T_p  T_p+1 ...  L_max-1
           ┌─────────────────────┬─────┬──────┬──────────────┐
prefill后：│K_0  K_1  K_2 ... K_Tp-1│  0  │  0   │    ...       │
           └─────────────────────┘     ↑
                已写入 T_p 个         cur_len=T_p（写入时 cache_position=0）

decode第1步后：
           │K_0  K_1 ... K_Tp-1  K_Tp│  0  │  0  │  ...         │
                                      ↑
                               新写入，cur_len → T_p+1

读取（decode 第 k 步）：
  full_k = k[:, :, :cur_len+1, :]   ← 切片只取有效范围，不含尾部零值
```

### 3.3 cur_len 推进时序

```
kv_cache.reset()              → cur_len = 0

prefill（一次写入 T_p 个 token）：
  写入位置 [0, T_p)           → 调用时 cache_position=cur_len=0
  kv_cache.cur_len = T_p      → 显式赋值为 T_p

decode step 1：
  写入位置 [T_p, T_p+1)       → cache_position = cur_len = T_p
  kv_cache.cur_len += 1       → cur_len = T_p+1

decode step 2：
  写入位置 [T_p+1, T_p+2)     → cache_position = cur_len = T_p+1
  kv_cache.cur_len += 1       → cur_len = T_p+2
  ...
```

**为什么 cur_len 放在 KVCache 而不是每层各自维护？**

所有层共享同一个 cur_len，`generate()` 里显式更新，状态变化一目了然，不需要 trace 进 cache 内部。对比 transformers StaticCache 把 `cumulative_length` 藏在 cache 对象里——封装更好，但状态不透明。inferlite 有意让状态可见，可读性优先。

## 4. Attention 读写逻辑

### 4.1 接口扩展

```python
# inferlite/model/attention.py - GQAAttention.forward 新签名
def forward(
    self,
    hidden_states: torch.Tensor,
    position_embeddings: tuple[torch.Tensor, torch.Tensor],  # (cos, sin)，由 Qwen3Model 统一计算
    layer_kv_cache: LayerKVCache | None = None,
    cache_position: int | None = None,  # = kv_cache.cur_len，由 generate loop 透传
) -> torch.Tensor:
```

**顺带优化**：M1 中每层 Attention 各自调用 `self.rotary_emb` 计算 cos/sin（28 层 = 28 次重复）。M2 把 `rotary_emb` 移到 `Qwen3Model.forward` 统一计算一次，传入所有层，与 transformers 对齐。

### 4.2 写入与读取

```python
if layer_kv_cache is not None:
    # 写入当前 token(s)：切片赋值，原地写入，不分配新内存
    # Prefill: T = T_p，cache_position = 0，写入 [0, T_p)
    # Decode:  T = 1， cache_position = cur_len，写入 [cur_len, cur_len+1)
    layer_kv_cache.k[:, :, cache_position:cache_position + T, :] = k
    layer_kv_cache.v[:, :, cache_position:cache_position + T, :] = v

    # 读取完整有效历史：只取 [0, cache_position+T) 范围，不含尾部零值
    k = layer_kv_cache.k[:, :, :cache_position + T, :]
    v = layer_kv_cache.v[:, :, :cache_position + T, :]
```

**为什么用切片而不是 transformers 的 `index_copy_`？**

| 方式 | 可读性 | MPS 兼容 | 性能 |
|------|--------|---------|------|
| 切片 `[:, :, pos:pos+T, :]` | ✅ 直接看到"写哪里、读哪里" | ✅ | ✅ decode 零内存分配 |
| `index_copy_` | ❌ 需要单独理解 API 语义 | ⚠️ MPS 有兼容问题 | ✅ 同样零分配 |
| `torch.cat` | ❌ 需要理解动态增长 | ✅ | ❌ 每步申请新内存 |

inferlite 优先可读性 + MPS 兼容，选切片。

### 4.3 Prefill vs Decode 时序对比

```
Prefill（T = T_p，cache_position = 0）：
─────────────────────────────────────────────────────────
  输入：hidden [B, T_p, H]
  q/k/v projection → q: [B, H_q, T_p, D]，k/v: [B, H_kv, T_p, D]
  q_norm / k_norm / RoPE（position_ids = [0, 1, ..., T_p-1]）
  写入：kv.k[:, :, 0:T_p, :] = k          ← 一次写满 T_p 个位置
  读取：full_k = kv.k[:, :, :T_p, :]      ← 读回 T_p 个（就是刚写的）
  Attention：Q[T_p × T_p] 带 causal mask（T > 1 触发）

Decode 第1步（T = 1，cache_position = T_p）：
─────────────────────────────────────────────────────────
  输入：hidden [B, 1, H]
  q/k/v projection → q: [B, H_q, 1, D]，k/v: [B, H_kv, 1, D]
  q_norm / k_norm / RoPE（position_ids = [[T_p]]，绝对位置）
  写入：kv.k[:, :, T_p:T_p+1, :] = k      ← 追加到第 T_p 个位置
  读取：full_k = kv.k[:, :, :T_p+1, :]    ← 读历史 T_p+1 个
  Attention：Q[1 × (T_p+1)] 无需 causal mask（T == 1 跳过）
```

### 4.4 causal mask：用 `T > 1` 判断

```python
# model/attention.py - GQAAttention.forward 中
T = hidden_states.shape[1]  # 当前输入序列长度
T_k = full_k.shape[2]       # K 的历史长度

if T > 1:  # prefill 需要因果 mask；decode 步 T=1，没有"未来"，不需要
    causal_mask = torch.triu(torch.ones(T, T_k, dtype=torch.bool, device=...), diagonal=1)
    attn_scores = attn_scores.masked_fill(causal_mask[None, None], float("-inf"))
```

**为什么不用 `if kv_cache is None` 判断？**

| 判断条件 | prefill 有 cache | prefill 无 cache | decode | 结论 |
|---------|----------------|----------------|--------|------|
| `if kv_cache is None` | ❌ 有 cache 时 mask 丢失 | ✅ | ✅ | **错误** |
| `if T > 1` | ✅ | ✅ | ✅ | **正确** |

causal mask 的需求来自"输入序列长度 > 1"（token 间有相互遮蔽），与 cache 是否存在无关。

## 5. Generate Loop 拆分

### 5.1 M2 完整数据流

```
generate()                         core.py
  │  kv_cache.reset()
  │
  ├─ Prefill：
  │    input_ids [B, T_p]
  │    position_ids [[0,1,...,T_p-1]]
  │    model(input_ids, position_ids, kv_cache)
  │      ↓
  │    kv_cache.cur_len = T_p        ← 显式赋值
  │    logits → 采样第一个生成 token
  │
  └─ Decode loop（step = 1..N）：
       cur_token [B, 1]
       position_ids [[cur_len]]      ← 绝对位置
       model(cur_token, position_ids, kv_cache)
         ↓
       kv_cache.cur_len += 1         ← 显式递增
       logits → 采样下一个 token → 继续/EOS 停止
```

**model 内部的数据流**（以 decode 步为例）：

```
model(cur_token [B,1], position_ids [[cur_len]], kv_cache)
  │
  ├─ embed_tokens → hidden [B, 1, H]
  ├─ position_embeddings = rotary_emb(hidden, position_ids)  ← 只算一次，传给所有层
  │
  └─ for i in range(L=28):
       DecoderLayer[i](hidden, position_embeddings,
                       layer_kv_cache=kv_cache.layers[i],
                       cache_position=kv_cache.cur_len)
         └─ GQAAttention.forward(...)
              ├─ q/k/v projection
              ├─ q_norm / k_norm / RoPE
              ├─ 写入：kv.k[:,:,cur_len:cur_len+1,:] = k
              ├─ 读取：full_k = kv.k[:,:,:cur_len+1,:]
              └─ Attention(q, full_k, full_v) → output
```

### 5.2 position_ids 必须用绝对位置

decode 步每次只处理 1 个 token，`position_ids = [[cur_len]]`（绝对位置），**不能写 `[[0]]`**。

**为什么 `[[0]]` 是沉默 bug？**

RoPE 的正确性依赖：`q_m · k_n` 应当编码相对距离 (m-n)。写 `[[0]]` 的后果：

```
decode step t：
  q 的 position = 0（错误，应为 cur_len = T_p + t）
  k_n 的 position = n（正确，从 cache 读取时已经正确编码）
  点积 q_0 · k_n 编码的相对距离 = (0 - n)，而非正确的 (T_p+t - n)
```

效果：每步 decode 都认为当前 token 在序列开头，RoPE 位置编码失效，生成质量严重下降——但**推理不报错**，这是最危险的 bug 类型。

### 5.3 向后兼容：kv_cache=None 退化为 M1

```python
def generate(..., kv_cache=None):
    if kv_cache is not None:
        # M2 路径
        ...
    else:
        # M1 路径：full forward（原逻辑完全不动）
        for _ in range(max_new_tokens):
            next_token = engine.step(input_ids)
            ...
```

**验收标准**：`generate(kv_cache=None)` 输出 `torch.equal` `generate(kv_cache=KVCache(...))` 输出。全部 95 个 M1 单测继续通过，无回归。

## 6. 与主流框架对比

| 维度 | transformers StaticCache | nano-vllm/vLLM | inferlite M2 |
|------|--------------------------|----------------|--------------|
| 内存策略 | 预分配连续 tensor，max_len 固定 | 全局预分配，block 粒度动态分配（PagedAttention） | 预分配连续 tensor，max_len 固定 |
| 写入方式 | `index_copy_` 原地写 | Triton kernel，slot 散列写 | 切片写入 `[:,:,pos:pos+T,:]` |
| 读取方式 | 返回完整 tensor（含尾部零值，需 mask） | FlashAttn + block_table 间接读 | 切片读有效范围 `[:,:,:pos+T,:]` |
| cur_len 管理 | 藏在 cache 对象（`cumulative_length`） | 调度器管理，外部不感知 | generate loop 显式维护 |
| position_ids | Model.forward 自动推算（隐式） | 调度器管理（隐式） | generate loop 显式传入 |
| 多序列 | 需 padding mask | 天然支持 continuous batching | 不支持（M3 范围） |
| torch.compile | ✅（StaticCache 专为此设计） | N/A（Triton kernel） | ❌（切片边界动态） |
| MPS 支持 | ⚠️（index_copy_ 有 fallback） | ❌（需 CUDA） | ✅ |
| 依赖 | 纯 PyTorch | Triton + FlashAttention | 纯 PyTorch |

**inferlite M2 的定位**：与 transformers StaticCache 最接近，但"切片写入替代 index_copy_"（MPS 兼容 + 更可读）、"cur_len 外置显式维护"（状态可见，教学目的）。

## 7. 实测性能

### 7.1 测试方法

```bash
# 脚本：scripts/bench_kv_cache.py
# 扫描不同 prompt 长度，对比 M1/M2 tok/s
uv run python scripts/bench_kv_cache.py \
  --model-dir ~/.cache/modelscope/hub/models/Qwen/Qwen3-0___6B \
  --device mps --dtype bf16 --gen-tokens 128
```

方法：用 pad_token 构造固定长度合成 prompt（控制变量），每种长度跑一次，`time.perf_counter` 计时，warmup 1 次（让 MPS 完成 JIT 编译）。

### 7.2 结果（Mac M3 Pro，MPS，bfloat16，gen_tokens=128）

| prompt_tokens | M1 tok/s | M2 tok/s | Speedup |
|--------------|---------|---------|---------|
| 32  | 13.8 | 24.9 | 1.80× |
| 64  | 12.7 | 28.9 | 2.27× |
| 128 |  9.6 | 25.7 | 2.67× |
| 256 |  6.2 | 24.1 | 3.91× |
| 512 |  3.3 | 24.1 | **7.36×** |

**规律**：
- M2 tok/s 基本稳定在 24-29（每步固定处理 1 个 token，O(T) 恒定）
- M1 tok/s 随 prompt 长度线性下降（O(T²) 逐渐显现）
- Speedup 随 prompt 长度线性增长，符合 O(T²)/O(T) = T 倍的理论预期

### 7.3 理论 ~600× vs 实测 7.36× 的差距

理论加速比假设 Attention 是唯一瓶颈。实测中，每步 decode 还需把 28 层权重（~1.2 GB）从内存读一遍：

```
单步 decode 总耗时 ≈ 权重 I/O 时间 + Attention 计算时间
                      ↑ M1/M2 共有       ↑ M2 省去了这部分
```

对 Qwen3-0.6B，`权重 I/O >> Attention 计算`，因此 Attention 节省被权重 I/O 掩盖，实测加速比远低于理论上限。

**KV Cache 的核心价值在**：
1. 长 prompt（prompt 越长，Attention 计算在总耗时中占比越高，加速比越大）
2. M3 Continuous Batching（多请求共享一次权重读取，吞吐量才真正拉开差距）

## 8. 踩坑与教训

| 坑 | 现象 | 根因 | 修复 |
|----|------|------|------|
| `position_ids = [[0]]` | 推理不报错，但生成质量差、输出混乱 | decode 步 RoPE 位置编码全在位置 0，相对距离信息错误 | `[[cur_len]]` 绝对位置 |
| causal mask 用 `if kv_cache is None` 判断 | prefill 有 cache 时 mask 丢失，logits 错误 | prefill/decode 的区分是 T>1，与有无 cache 无关 | 改为 `if T > 1` |
| `reset()` 是否要清零 tensor | 不清零也完全正确 | prefill 覆盖写入 [0, T_p)，旧数据不影响 | 只清 `cur_len`，无需 `fill_(0)` |
| `AutoTokenizer.from_pretrained` 报 HFValidationError | 本地路径被当作 HF repo id 验证，报错 | transformers 5.x 对本地路径的校验更严格 | 加 `local_files_only=True` |
| ModelScope 缓存路径 `.` 变 `___` | 手动拼路径时找不到文件 | ModelScope 对文件系统特殊字符的安全替换（`0.6B` → `0___6B`） | 用 `find` 确认实际路径 |

## 10. 下一步：M3 Continuous Batching

M2 留下三个接口桩，M3 在此基础上生长：

**1. KVCache 是单序列设计**

`cur_len` 只有一个，意味着 batch 内所有序列同步推进。M3 需要每个请求有独立的 `cur_len`，或改为 PagedAttention 的 `block_table` 管理（逻辑地址→物理 block 映射）。

**2. generate() 是单请求循环**

M3 的 `FCFSScheduler` 把多个请求合并进同一个 `forward`，共享一次权重读取。这是吞吐量的真正来源：

```
单序列 KV Cache：省去的是 Attention 的 O(T²) 计算
Continuous Batching：省去的是每请求独立读一遍权重的 I/O
```

两者叠加才能接近生产级 LLM 推理服务的吞吐量。

**3. device/dtype 已就绪**

M2 T5 完成的 `--device mps/cuda` + `bf16` 支持是 M3 的先决条件，可直接复用。
