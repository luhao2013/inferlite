# M3 — Continuous Batching：从单请求 KV Cache 到多请求 Decode 调度

> **状态**：规划中
> **前置**：M2 — 单序列静态 KV Cache
> **主战场**：L3 Engine / Scheduler
> **核心目标**：把 M2 的单请求 decode 扩展为多请求 decode batching：每生成一个 token 重新组一次 batch，让多个请求共享一次模型 forward。

## 1. Continuous Batching 到底是什么

一句话：

> **Continuous batching = 每生成一个 token，都重新决定这一轮 batch 里有哪些请求。**

它也叫：

```text
iteration-level scheduling
```

也就是：

```text
按 token iteration 调度，而不是按 request 生命周期调度。
```

---

## 2. Static Batching vs Continuous Batching

### 2.1 Static batching：batch 成员固定

假设有 3 个请求：

```text
reqA: 需要生成 2 个 token
reqB: 需要生成 5 个 token
reqC: 需要生成 3 个 token
```

static batching 会在开始时组一个固定 batch：

```text
batch = [A, B, C]
```

然后这个 batch 一起跑到最长请求结束：

```text
time →
step1   step2   step3   step4   step5
 A       A       pad     pad     pad
 B       B       B       B       B
 C       C       C       pad     pad
```

问题：

- A 第 2 步已经完成，但要等 B 第 5 步结束才能整体返回。
- C 第 3 步已经完成，也要陪跑。
- 新来的请求 D 不能加入，只能等整个 batch 完成。

这就是 Head-of-Line Blocking / 木桶效应。

### 2.2 Continuous batching：每一步 batch 成员都可以变化

同样的请求，continuous batching 会在每个 decode step 后重新调度：

```text
time →
step1   step2   step3   step4   step5
 A       A       -       -       -
 B       B       B       B       B
 C       C       C       -       -
 -       -       D       D       D
```

核心变化：

```text
短请求完成后立即出队
新请求可以在下一个 iteration 加入
batch 不需要固定到所有请求结束
```

### 2.3 一个更直观的类比

Static batching 像：

```text
一车人从起点坐到终点，中途不准上下车。
```

Continuous batching 像：

```text
公交车每站都可以有人下车、有人上车，车一直向前跑。
```

对应 LLM：

```text
每个 decode step 都可以有人完成、有人加入，GPU 一直跑。
```

## 3. 为什么 LLM Decode 天然适合 Continuous Batching

LLM 推理分两段：

```text
Prefill：一次性处理完整 prompt
Decode：每次只处理上一步生成的 1 个 token
```

对于多个请求，prefill 长度可能差异很大：

```text
reqA prompt_len = 100
reqB prompt_len = 300
reqC prompt_len = 80
```

但进入 decode 后，每个请求每一步都是：

```text
reqA 当前输入: [1 token]
reqB 当前输入: [1 token]
reqC 当前输入: [1 token]
```

所以 decode 可以天然拼成：

```text
input_ids shape = [B, 1]
```

历史上下文不再靠 `input_ids` 携带，而是保存在各自的 KV Cache 里。

这就是 M2 对 M3 的关键意义。

## 4. M2 在整个项目中的位置

inferlite 项目四层抽象：

```text
L4 Server  : HTTP / SSE / OpenAI API
L3 Engine  : scheduler / batching / sampling
L2 Memory  : KV Cache / PagedAttention / Prefix Cache
L1 Model   : Transformer forward
```

M1 打通：

```text
L1 Model + 最小 L3 Engine
模型能 forward，CLI 能出字，但每步重算完整历史
```

M2 打通：

```text
L2 Memory 的第一个版本：单请求静态 KV Cache
把推理拆成 prefill + decode
让 decode 每步只输入 1 个 token
```

M3 要打通：

```text
L3 Scheduler：多请求 continuous batching
```

M4 再升级：

```text
L2 Memory：PagedAttention，解决 M3 固定 slot 的空间浪费
```

### 4.1 没有 M2，就很难做 M3

如果没有 KV Cache，每个请求 decode 时仍然要输入完整历史：

```text
reqA: [1, 100]
reqB: [1, 300]
reqC: [1, 80]
```

这样又会回到 padding / full-sequence batching 的复杂问题。

有了 M2 后，每个请求 decode 统一变成：

```text
reqA: last_token [1, 1] + KV slotA
reqB: last_token [1, 1] + KV slotB
reqC: last_token [1, 1] + KV slotC
```

所以 M3 可以把 `last_token` 拼成 `[B, 1]`，同时传入每个请求自己的 KV 位置。

一句话：

> **M2 是单请求 KV Cache 地基；M3 是在这个地基上做多请求调度；M4 是把这个地基的内存管理从固定 slot 升级成分页。**

### 4.2 M2 到 M3：从无 batch 到选择 continuous batching

M2 只有单请求 generate：

```text
request A: prefill + decode_1 + decode_2 + ...
request B: prefill + decode_1 + decode_2 + ...
```

如果有多个请求，M2 只能串行跑：

```text
先完整跑完 A
再完整跑完 B
再完整跑完 C
```

因此 M3 的第一性问题不是“KV Cache 怎么更省显存”，而是：

```text
LLM 推理引擎要不要支持 batch？如果支持，选择哪种 batch？
```

对普通分类 / embedding 模型，一个请求通常一次 forward 就结束：

```text
request = one forward
```

这种场景 static batching 足够：

```text
收集 N 个请求 → 拼成 batch → 一次 forward → 全部返回
```

但 LLM 生成请求不是一次 forward，而是：

```text
request = prefill + decode_1 + decode_2 + ... + decode_N
```

其中 `N` 不确定：有的请求很快遇到 EOS，有的请求会生成很长。也就是说，LLM 的关键不定长不只是 prompt 长度不同，更是**生成长度 / decode iteration 数不同**。

如果 M3 选择 static batching，batch 成员会固定到最长请求结束：

```text
time →
step1   step2   step3   step4   step5
 A       A       pad     pad     pad
 B       B       B       B       B
 C       C       C       pad     pad
```

这会造成：

```text
短请求陪长请求跑空步
新请求不能及时加入
batch slot 被 finished 请求占着
用户延迟变高，GPU 利用率下降
```

所以 M3 选择 continuous batching：

```text
每个 decode iteration 重新组一次 batch
完成的请求立即出队
新请求在下一个 iteration 边界加入
```

也就是：

```text
time →
step1   step2   step3   step4   step5
 A       A       -       -       -
 B       B       B       B       B
 C       C       C       -       -
 -       -       D       D       D
```

因此，M3 的技术选型可以总结为：

> **M2 → M3 是从“无 batch 的单请求推理”进入“有 batch 的多请求推理”；因为 LLM 生成长度不定，static batching 会产生木桶效应，所以 M3 选择 continuous batching 作为 batch 策略。**

## 6. 技术调研：其他框架怎么做

### 6.1 Orca：提出 iteration-level scheduling

Orca OSDI'22 指出，传统 inference serving 不适合自回归生成这种 multi-iteration workload：当前 batch 不能动态改变，早完成请求不能及时返回，新请求必须等当前 batch 完全结束。Orca 提出 iteration-level scheduling，把调度粒度从 request 改成 iteration，每次只执行 batch 的一个模型迭代。[^orca]

Orca 还有一个关键词：

```text
selective batching
```

它不是把 Transformer 所有操作都无脑 batch，而是选择适合 batch 的部分进行 batch。

inferlite M3 的裁剪正是教学版 selective batching：

```text
prefill 逐条做
decode 组 batch
```

### 6.2 FriendliAI：iteration batching / continuous batching

FriendliAI 将这个思想称为 iteration batching / continuous batching：batch 在运行过程中动态改变，已完成请求可以立即返回，新请求可以加入后续迭代。[^friendli]

这说明 continuous batching 的本质不是“等人凑满 batch”，而是：

```text
batch membership 在 iteration 边界连续流动
```

### 6.3 vLLM：continuous batching + PagedAttention

vLLM V1 中，Engine Core 负责 scheduler、KV cache 管理和模型执行协调；API Server 负责 HTTP 和输入处理；GPU Worker 负责模型 forward 和显存管理。[^vllm-arch]

生产版 vLLM 大致流程：

```python
while True:
    scheduled = scheduler.schedule()
    allocate_kv_blocks_for_new_requests(scheduled)
    inputs = model_runner.prepare_inputs(
        input_ids,
        positions,
        slot_mapping,
        block_tables,
        seq_lens,
    )
    logits = model_runner.execute_model(inputs)
    next_tokens = sampler(logits)
    scheduler.update(next_tokens)
    free_blocks_for_finished_requests()
```

vLLM 比 M3 多很多能力：

- PagedAttention：KV Cache 按 block/page 分配。
- Chunked prefill：长 prompt 分块，避免 prefill 阻塞 decode。
- Prefix cache：相同前缀复用 KV。
- CUDA Graph / fused kernel / FlashAttention。
- 多进程、多 GPU、TP/PP/DP。

### 6.4 SGLang：continuous batching + prefix/radix cache

SGLang 也做 continuous batching，但更强调 prefix 复用，例如 RadixAttention / radix tree cache。

它会让多个请求共享公共前缀的 KV：

```text
system prompt
  ├── user A branch
  └── user B branch
```

这类 prefix/session cache 不属于 M3，放到 M5 更合适。

### 6.5 nano-vLLM：教学版 vLLM

nano-vLLM 通常保留：

```text
waiting queue
running list
block manager
PagedAttention 简化版
```

inferlite M3 比 nano-vLLM 再裁剪一层：

```text
不用 block manager
先用固定 slot manager
```

目的是把 M3 聚焦在 continuous batching 本身，而不是提前进入 PagedAttention。

## 7. inferlite M3 的目标与非目标

### 7.1 一句话目标

> **M3 实现教学版 continuous batching：每个 decode iteration 重新组 batch，running 请求共享一次模型 forward；KV Cache 用固定 slot 方案，每个请求维护自己的 `slot_id + seq_len`。**

### 7.2 能力范围

| 能力 | M3 是否支持 | 说明 |
|---|---:|---|
| 单请求 M2 generate | ✅ | 保持兼容 |
| 多请求提交 | ✅ | 支持 waiting queue |
| waiting / running / finished 三队列 | ✅ | L3 调度核心 |
| prefill 逐条做 | ✅ | 避免 batch prefill padding 复杂度 |
| decode batch | ✅ | M3 核心能力 |
| 每请求独立 cache length | ✅ | `seq_lens[slot]` |
| 固定 slot-based KV Cache | ✅ | M3 教学版内存方案 |
| 不同请求 EOS 早停 | ✅ | done 请求立即出队 |
| 新请求中途加入 | ✅ 简化支持 | decode step 边界 admit |
| batch prefill | ❌ | 留后续 |
| chunked prefill | ❌ | 留 M11 |
| PagedAttention | ❌ | 留 M4 |
| Prefix / session cache | ❌ | 留 M5 |
| 有状态多轮 KV 复用 | ❌ | 留 M5 Prefix / Session Cache |
| 请求抢占 / preemption | ❌ | 留后续 |
| batching window / 等待凑 batch | ❌ | M3 不做时间窗口策略 |

## 8. 为什么 M3 只 batch decode，不 batch prefill

### 8.1 Batch prefill 的 padding 计算浪费严重

假设：

```text
reqA prompt_len = 100
reqB prompt_len = 300
reqC prompt_len = 80
```

naive batch prefill 需要 padding 到最大长度 300：

```text
reqA: 100 token + 200 pad
reqB: 300 token
reqC:  80 token + 220 pad
```

真实 token 数：

```text
100 + 300 + 80 = 480
```

batch 后计算 token 数：

```text
3 × 300 = 900
```

浪费：

```text
420 / 900 ≈ 46.7%
```

而且 prefill padding 浪费的不只是 KV 空间，还包括：

```text
embedding / qkv projection / RMSNorm / RoPE / Attention / MLP / lm_head
```

prefill attention 还是 `T × T`，长度差异越大，padding 浪费越严重。

### 8.2 Decode batch 的浪费小得多

decode 每个请求每步都是 1 token：

```text
input_ids shape = [B, 1]
```

没有 query token padding。

但是 KV 历史长度不同，M3 PyTorch 版会按 batch 内最大长度 gather：

```text
reqA seq_len = 100
reqB seq_len = 300
reqC seq_len = 80
max_len = 300
```

短请求会读到并 mask 掉尾部无效 KV，带来 O(T) attention 浪费。

相比 prefill 的 O(T²) padding 浪费，这个代价可接受。真正消除它留给 M4 PagedAttention。

## 9. 请求组 batch 是否要等

M3 不做“等待凑满 batch”。

M3 采用 opportunistic batching：

> **每个 decode iteration 边界，调度器拿当前已经 ready 的 running 请求组成 batch；有多少跑多少，不为了凑满 batch 无限等待。**

如果当前只有 1 个请求：

```text
batch size = 1
```

如果当前有 8 个请求：

```text
batch size = 8
```

新请求最多等待到下一个 iteration 边界。

M3 不做：

| 策略 | M3 是否做 |
|---|---:|
| batching window，例如等 5ms 凑 batch | ❌ |
| max waiting time | ❌ |
| SLO-aware scheduling | ❌ |
| priority queue | ❌ |
| preemption | ❌ |

原因：M3 没有 HTTP server，没有真实在线到达时间；教学目标是讲清 iteration-level scheduling，不引入 serving SLO。

## 10. M3 是否需要固定大小 KV Cache

### 10.1 结论

Continuous batching 本身不要求固定大小 KV Cache。

但 inferlite M3 第一版建议使用固定大小 slot-based KV Cache：

```python
BatchedKVCache(
    max_num_slots=S,
    max_seq_len=L,
)
```

K/V 形状：

```python
k_cache.shape = [S, n_kv_heads, L, head_dim]
v_cache.shape = [S, n_kv_heads, L, head_dim]
```

其中：

| 名称 | 含义 |
|---|---|
| `S = max_num_slots` | 最多同时 running 的请求数 |
| `L = max_seq_len` | 每个请求最多写入多少 token 的 KV |
| `seq_lens[slot]` | 某个 slot 当前真实有效长度 |
| `occupied[slot]` | slot 是否被占用 |

### 10.2 为什么 M3 用固定 slot

因为 M3 要学习的是：

```text
scheduler 如何每步重新组 batch
```

不是：

```text
显存 allocator / block manager 怎么设计
```

固定 slot 最容易解释：

```text
一个 running 请求 = 一个完整 slot
slot_id = 请求的物理 KV 地址
seq_len = 请求当前有效长度
```

### 10.3 固定 slot 的代价

显存空间浪费。

例如：

```text
max_num_slots = 4
max_seq_len = 1024
```

当前请求：

```text
reqA = 100
reqB = 300
reqC = 80
```

有效 KV token：

```text
480
```

预留 KV token：

```text
4 × 1024 = 4096
```

空间利用率很低。

这正是 M4 PagedAttention 要解决的问题。

## 11. `max_num_slots`、`cache_slots`、waiting queue 的区别

这三个概念不能混：

| 概念 | 含义 |
|---|---|
| `max_num_slots` / `max_num_seqs` | BatchedKVCache 预分配了多少个 slot；限制同时 running 的最大请求数 |
| `cache_slots` | 当前 decode step 实际参与 batch 的 slot id 列表，shape `[B]` |
| waiting queue 容量 | 服务层最多能排队多少请求，可以大于 `max_num_slots` |

例子：

```text
max_num_slots = 8
```

当前 slot 状态：

```text
slot 0: reqA, seq_len=100
slot 1: free
slot 2: reqB, seq_len=300
slot 3: reqC, seq_len=80
slot 4: free
slot 5: reqD, seq_len=120
slot 6: free
slot 7: free
```

当前 running 请求是 A/B/C/D，所以当前 decode step：

```python
cache_slots = torch.tensor([0, 2, 3, 5])
cache_positions = torch.tensor([100, 300, 80, 120])
```

结论：

```text
max_num_slots 是容量
cache_slots 是当前 step 用到哪些 slot 的索引
```

M3 简化版可以让：

```text
max_num_slots == max_num_seqs_per_step
```

也就是有几个 slot，最多 batch 几条 decode 请求。

## 12. KV Cache 是否有过期概念

M3 有释放，没有 session 级 TTL。

M3 的 KV 生命周期绑定 active request：

```text
submit request
  ↓
alloc slot
  ↓
prefill + decode
  ↓
EOS / max_new_tokens / error / cancel
  ↓
free slot
```

M3 不做跨请求 KV 复用，因此不需要 TTL / LRU / eviction。

| Cache 类型 | 是否跨请求 | 是否需要 TTL | 归属 |
|---|---:|---:|---|
| Request KV Cache | 否 | 不需要，结束即释放 | M3 |
| Session KV Cache | 是 | 需要 | M5+ |
| Prefix Cache | 是 | 需要 eviction / refcount / LRU | M5 / M4 后更自然 |

M3 的 slot 不是“可复用历史上下文缓存”，而是 active request 的工作内存。

## 13. M3 的 HBM / gather / attention 计算方式

### 13.1 KV 逻辑独立，物理上统一预分配

M3 不是每个请求单独创建一个 `KVCache` 对象：

```python
reqA.kv_cache = KVCache(...)
reqB.kv_cache = KVCache(...)
```

而是创建一个统一的 `BatchedKVCache`：

```python
k_cache.shape = [max_num_slots, n_kv_heads, max_seq_len, head_dim]
v_cache.shape = [max_num_slots, n_kv_heads, max_seq_len, head_dim]
```

逻辑上：

```text
reqA -> slot0
reqB -> slot2
reqC -> slot3
```

物理上：

```text
一整块 HBM / MPS unified memory 中的 dense tensor
```

### 13.2 Query 是 batch 维度拼接

decode 输入：

```python
input_ids = torch.cat([
    reqA.last_token,
    reqB.last_token,
    reqC.last_token,
], dim=0)  # [B, 1]
```

经过 embedding / q_proj：

```text
q.shape = [B, n_q, 1, D]
```

这就是多个请求 query 在 batch 维度拼起来。

### 13.3 KV 在 M3 PyTorch 版会 gather 成临时 batched tensor

当前 batch：

```python
cache_slots = [0, 2, 3]
cache_positions = [100, 300, 80]
max_len = 301
```

从 cache 中取：

```python
k = k_cache[cache_slots, :, :max_len, :]
v = v_cache[cache_slots, :, :max_len, :]
```

得到：

```text
k.shape = [B, n_kv, max_len, D]
v.shape = [B, n_kv, max_len, D]
```

这一步在 PyTorch 教学版中会 materialize 一个临时 dense batched KV tensor。

注意：这是 **batch 维度 gather**，不是把请求沿 sequence 维度拼成一个长序列。

正确：

```text
k[0] = slot0[0:301]
k[1] = slot2[0:301]
k[2] = slot3[0:301]
```

错误：

```text
[slot0 KV][slot2 KV][slot3 KV] 拼成一个超长 sequence
```

后者会导致跨请求互相 attend，是错误的。

### 13.4 Attention 是逐 batch row 独立算

```python
scores = q @ k.transpose(-2, -1)
```

实际含义：

```text
scores[0] = qA @ k_slot0.T
scores[1] = qB @ k_slot2.T
scores[2] = qC @ k_slot3.T
```

不会发生：

```text
qA attends to reqB 的 KV
```

### 13.5 与生产 PagedAttention 的区别

M3 PyTorch 教学版：

```text
HBM 里先 gather/materialize [B, max_len, D] 的临时 KV
再交给普通 PyTorch attention kernel
```

生产 PagedAttention：

```text
KV 以 blocks 形式常驻 HBM
kernel 接收 block_table / seq_lens
kernel 内部按 tile 从 HBM 读 K/V 到 shared memory / registers
不在 HBM 中 materialize 完整 [B, max_len, D] 临时 KV
```

所以：

```text
M3 选简单可读的 gather + dense attention
M4 再做 block_table + kernel 内按块读取
```

## 14. Decode Attention 的 mask

假设：

```text
reqA: slot=0, cache_position=100
reqB: slot=2, cache_position=300
reqC: slot=3, cache_position=80
```

当前 decode token 写入后：

```text
reqA valid KV = 0..100
reqB valid KV = 0..300
reqC valid KV = 0..80
```

读取 batch 内最大长度：

```python
valid_lens = cache_positions + 1
max_len = int(valid_lens.max().item())
```

构造 mask：

```python
positions = torch.arange(max_len, device=device)
visible = positions[None, :] < valid_lens[:, None]
# 等价于：positions[None, :] <= cache_positions[:, None]
```

应用到 scores：

```python
scores = torch.matmul(q, k.transpose(2, 3)) * scaling
# scores: [B, n_q, 1, max_len]

scores = scores.masked_fill(
    ~visible[:, None, None, :],
    torch.finfo(scores.dtype).min,
)
```

含义：

```text
reqA 只能看 KV position 0..100
reqB 只能看 KV position 0..300
reqC 只能看 KV position 0..80
```

当前 query 会 attend to 自己的 K，这是正确的 causal self-attention：

```text
q_t attends to k_0..k_t
不能看 k_{t+1}...
```

## 15. 核心数据结构

### 15.1 RequestState

```python
@dataclass
class RequestState:
    request_id: str
    prompt_ids: torch.Tensor        # [1, T_p]
    generated_ids: list[torch.Tensor]

    max_new_tokens: int
    eos_token_id: int | None

    status: RequestStatus           # waiting / running / finished / cancelled

    slot_id: int | None             # 分配到哪个 KV slot
    seq_len: int                    # 当前 KV Cache 已写入长度
    num_generated: int              # 已采样 token 数

    last_token: torch.Tensor | None # 下一次 decode 要输入的 token
```

关键语义：

```text
seq_len    = KV Cache 已写入长度
last_token = 下一次 decode 要输入，但还没写进 cache 的 token
```

prefill 后：

```text
prompt_len = 100
seq_len = 100
prefill logits 采样得到 token_x
last_token = token_x
```

下一次 decode：

```text
输入 token_x
写入 cache[100]
seq_len = 101
采样得到 token_y
last_token = token_y
```

### 15.2 BatchedKVCache

```python
class BatchedKVCache:
    layers: list[LayerKVCache]
    seq_lens: torch.Tensor      # [max_num_slots]
    occupied: torch.Tensor      # [max_num_slots]
```

每层：

```python
layer.k.shape = [max_num_slots, n_kv_heads, max_seq_len, head_dim]
layer.v.shape = [max_num_slots, n_kv_heads, max_seq_len, head_dim]
```

### 15.3 FCFSScheduler

```python
class FCFSScheduler:
    waiting: deque[RequestState]
    running: dict[str, RequestState]
    finished: dict[str, RequestState]

    def submit(req): ...
    def admit(): ...
    def mark_finished(req): ...
```

不变式：

```text
len(waiting) + len(running) + len(finished) == total_requests
```

## 16. M3 执行流程

### 16.1 主循环

```python
while scheduler.has_unfinished():
    # 1. 有空 slot 就从 waiting admit 新请求
    admit_until_full()

    # 2. 新 admitted 请求逐条 prefill
    prefill_new_requests_one_by_one()

    # 3. 当前 running 请求做一次 batched decode
    decode_one_iteration_for_running_batch()

    # 4. 完成请求出队并释放 slot
    finish_done_requests()
```

### 16.2 单条 Prefill

```python
req = scheduler.admit()
slot = kv_cache.alloc_slot(req.request_id)

T_p = req.prompt_ids.shape[1]
position_ids = torch.arange(T_p).unsqueeze(0)

logits = model(
    input_ids=req.prompt_ids,       # [1, T_p]
    position_ids=position_ids,      # [1, T_p]
    kv_cache=kv_cache,
    cache_slots=torch.tensor([slot]),
    cache_positions=torch.tensor([0]),
)

req.slot_id = slot
req.seq_len = T_p
req.last_token = sampler(logits[:, -1, :])
req.num_generated = 1
req.status = RUNNING
```

### 16.3 Batched Decode

```python
running_reqs = list(scheduler.running.values())

input_ids = torch.cat([
    req.last_token for req in running_reqs
], dim=0)  # [B, 1]

cache_slots = torch.tensor([
    req.slot_id for req in running_reqs
])  # [B]

cache_positions = torch.tensor([
    req.seq_len for req in running_reqs
])  # [B]

logits = model(
    input_ids=input_ids,
    position_ids=cache_positions[:, None],
    kv_cache=kv_cache,
    cache_slots=cache_slots,
    cache_positions=cache_positions,
)

next_tokens = sampler(logits[:, -1, :])

for req, next_token in zip(running_reqs, next_tokens):
    req.seq_len += 1
    req.num_generated += 1
    req.last_token = next_token

    if next_token == req.eos_token_id or req.num_generated >= req.max_new_tokens:
        scheduler.mark_finished(req)
        kv_cache.free_slot(req.slot_id)
```

## 17. 与 Orca / vLLM / SGLang / nano-vLLM 的差异

| 维度 | Orca | vLLM | SGLang | nano-vLLM | inferlite M3 |
|---|---|---|---|---|---|
| 核心思想 | iteration-level scheduling | continuous batching + PagedAttention | continuous batching + prefix/radix cache | vLLM 简化版 | continuous batching 教学最小版 |
| batch 何时更新 | 每个 iteration | 每个 scheduler step | 每个 scheduler step | 每个 step | 每个 decode step |
| 新请求何时加入 | 下一 iteration | scheduler step | scheduler step | scheduler step | decode step 边界 |
| 完成请求何时退出 | 下一 iteration | 释放 block | 释放 cache / block | 释放 block | 释放 slot |
| KV 管理 | 系统级 | block/page | radix + block | block | fixed slot |
| prefill | selective batching | token budget + chunked prefill | prefix-aware prefill | 简化 | 逐条 prefill |
| decode | batch | batch | batch | batch | batch |
| 时间策略 | serving 系统考虑 SLO | 有复杂调度参数 | 有 serving 策略 | 简化 | 不等人凑 batch |
| cache 过期 | 系统策略 | refcount / eviction / prefix cache | prefix eviction | 简化 | active request 结束即释放 |
| 目标 | 生产系统 | 生产系统 | 生产系统 | 小型教学实现 | 学清楚 continuous batching |

## 18. M3 与 M4 的边界

M3：

```text
slot-based KV Cache
一个请求占一个完整固定 slot
decode 时 gather 出临时 dense batched KV
实现简单，但浪费显存空间和部分 gather/attention 计算
```

M4：

```text
PagedAttention
一个请求由多个 block 组成
按需分配 KV block
通过 block_table 在 kernel 内读取 KV tile
减少显存浪费，避免完整 dense gather
```

M3 解决：

```text
多请求 decode 的权重 I/O 重复
```

M4 解决：

```text
KV Cache 空间浪费与碎片化
```

## 19. 验收目标

### 19.1 正确性

- 单请求 M2 路径保持不变。
- 多请求 batched decode 输出与逐条串行 M2 输出完全一致。
- 不同 prompt 长度的请求可以同 batch decode。
- 不同生成长度 / EOS 早停请求可以提前出队。
- slot 释放后可以复用，旧 KV 不会泄漏。
- `seq_lens[slot]` 独立维护，互不影响。

### 19.2 性能

目标：

```text
8 并发请求聚合吞吐 ≥ 串行 M2 ×4
```

这个目标来自：

```text
多个请求共享一次模型权重读取 / forward launch
```

Mac MPS 可先验证功能；最终性能数最好在 GPU 上补测。

## 20. 本版设计决策

| 决策 | 结论 | 原因 |
|---|---|---|
| Prefill 是否 batch | 否 | padding 计算浪费严重，且引入复杂 mask / chunked prefill |
| Decode 是否 batch | 是 | 每请求每步都是 1 token，天然适合组 batch |
| 是否等待凑 batch | 否 | M3 不做 batching window；iteration 边界即刻调度 |
| KV Cache 是否固定大小 | 是 | M3 教学版用 `max_num_slots × max_seq_len` 固定 slot |
| `cache_slots` 是什么 | 当前 step 的 slot id 列表 | 容量是 `max_num_slots`，不是 `cache_slots` |
| Decode KV 是否按 max_len gather | 是 | PyTorch 教学版实现简单，用 per-row mask 保证正确 |
| 是否在 HBM materialize 临时 KV | 是 | M3 用 gather + dense attention；M4 再避免 |
| 每请求是否独占 slot | 是 | M2 静态 KV Cache 的自然扩展 |
| KV 是否有 TTL | 无 session TTL | request 完成 / 取消 / error 即释放 slot |
| 是否支持 PagedAttention | 否 | 留 M4 |
| 是否支持有状态多轮 KV 复用 | 否 | 留 M5 Prefix / Session Cache |

---

## 21. 一句话总结

> **M3 的本质是：把请求看成一串 token-level 小任务，每生成一个 token 就重新组一次 batch。M2 让每个请求的历史从长 `input_ids` 转移到 KV Cache；M3 在此基础上用固定 slot 保存每个 running 请求的 KV，用 `cache_slots + seq_lens` 描述当前 batch 的 KV 位置，decode 时 gather 出临时 batched KV 并用 per-row mask 计算 attention。它不等人凑 batch，不做 session TTL，不解决 KV 空间浪费；这些生产能力分别留给 M4 PagedAttention 和 M5 Prefix/Session Cache。**
