    M2 — 单序列静态 KV Cache#

    完成日期：2026-06-29 设计文档（含完整 ADR）：m2-kv-cache-design.md 实测结果归档：bench/results/2026-06-29-m2-kv-cache-mps-bf16.md

    范围与边界#

    M2 实现的是最简可用的 KV Cache，刻意控制范围，聚焦于核心机制的理解与正确实现。明确不在范围内的内容：

    限制 当前状态 解决方向
    只支持单序列（batch=1） cur_len 全局唯一，所有请求必须同步推进 M3：每请求独立 cur_len，或 PagedAttention
    静态预分配，长度固定 max_seq_len 在 generate 前确定，超出报 IndexError 动态扩容或 PagedAttention 按需分配
    无内存复用 每次 generate 分配新 cache，结束后释放 M3 维护全局 cache pool
    不支持 prefix caching 不同请求的相同 prompt 前缀各自重算 prefix cache / radix tree
    不支持 torch.compile 切片边界 cur_len 动态变化，无法静态编译 改用固定 shape + mask 方案

    这是教学级实现，适合理解 KV Cache 的核心机制。生产级需求（高吞吐、多并发、长上下文）对应 M3（Continuous Batching）及后续阶段。

    1. 为什么需要 KV Cache#

    1.1 LLM 推理 = 一次 prefill + N 次 decode#

    LLM 生成文本的过程分两个阶段：

    Prefill：把用户输入的 prompt（假设 T_p 个 token）一次性全部送入模型，并行计算所有 token 的 Attention，得到第一个生成 token 的 logits。这一步是必须的，所有 token 之间要相互计算 Attention。

    Decode：每步只生成一个 token，然后把这个 token 追加到序列末尾，再过一遍完整模型，生成下一个 token。如此循环 N 次，直到生成完毕或遇到 EOS。

    没有 KV Cache 时（M1 的做法），decode 第 k 步的输入是完整历史序列：

    decode 第 1 步：输入 [prompt + token_1]，长度 T_p+1
decode 第 2 步：输入 [prompt + token_1 + token_2]，长度 T_p+2
...
decode 第 N 步：输入 [prompt + token_1 + ... + token_N]，长度 T_p+N

    1.2 重复计算在哪里#

    每步 decode 过 Attention 时，要计算序列中所有 token 的 K（Key）和 V（Value）。但历史 token 的 K/V 每次都一样——第 k 步和第 k-1 步相比，只多了最新生成的那一个 token，其余所有 token 的 K/V 完全相同。

    这是 LLM 推理的根本浪费：N 步 decode 重复计算了同一份历史 K/V 共 N 次。

    LLM 推理的计算开销高度集中于 Attention 和 FFN 两个模块，二者合计占推理总延迟的 94%～96%（数据来自中科院计算所对 Llama-3-8B、Qwen2.5-7B/32B 的系统性实测，arXiv 2512.01644）。在 Decode 阶段，随着上下文变长，Attention 的开销占比持续上升：

    场景 典型上下文（1K–8K） 超长上下文（>16K）
    小模型（8B） Decode Attention 主导 Attention 主导
    大模型（32B）Decode FFN 主导（固定开销大） Attention 主导

    根本原因在于复杂度的差异：Decode 阶段 FFN 的计算量是 O(1)（每步固定），而 Attention 由于需要遍历完整 KV Cache，代价是 O(T)——随上下文线性增长。没有 KV Cache 时退化为 O(T²)，每步 decode 的 Attention 计算量：

    单步 decode（无 KV Cache）：Attention 计算量 ≈ O(T²)   T = 当前序列长度
N 步总量 ≈ O(N · T²)                                    T_p=100, N=500 时约 1.8 亿次运算

    1.3 KV Cache：空间换时间#

    解法直接：把每步 decode 计算出来的 K/V 存起来，下一步直接读取，不再重算。

    有 KV Cache 后，decode 第 k 步：
  1. 只计算当前 1 个 token 的 Q（Query）
  2. 从 cache 里读取所有历史 K/V
  3. 当前 Q 与历史 K/V 做 Attention，计算量 O(T)（1×T 向量乘，而非 T×T）
  4. 把当前 token 的 K/V 写入 cache

    N 步总计算量降至 O(N · T)，比无 cache 的 O(N · T²) 少了 T 倍。

    为什么只 cache K 和 V，不 cache Q？

    • Q（Query）= "当前 token 想查什么"：每步生成的新 token 不同，Q 每次都要重新计算，无法复用。
    • K/V（Key/Value）= "历史 token 的内容"：由 causal mask 的性质保证，位置 t 的 K/V 只取决于 0..t 的 token，不受后续新 token 影响，在整个 decode 过程中永远不变。

    这是 KV Cache 名字的由来：只 cache K 和 V，不 cache Q。

    1.4 代价：显存占用和内存带宽压力#

    KV Cache 是典型的空间换时间，代价有两个：

    显存占用：需要为每一层、每一个位置预留 K/V 的存储空间。Qwen3-0.6B 在 bf16 精度下，L_max=1024 时占用约 58 MB。这个数字随序列长度线性增长（L_max=4096 时 ~233 MB）。

    内存带宽压力：每步 decode 需要把整个 cache 读出来参与 Attention 计算。随着序列变长，每步读取的数据量也线性增长，可能成为新的瓶颈（compute-bound → memory-bound）。

    1.5 针对 KV Cache 代价的优化：GQA#

    缩减 KV Cache 显存最直接的方式是减少 K/V head 数。标准 Multi-Head Attention（MHA）中 K/V head 数等于 Q head 数，GQA（Grouped Query Attention）让多个 Q head 共享同一组 K/V head：

    MHA：H_q = H_kv = 16  →  KV Cache 每层 16 head × head_dim
GQA：H_q = 16，H_kv = 8  →  KV Cache 每层 8 head × head_dim（减半）

    Qwen3-0.6B 采用 GQA（H_q=16，H_kv=8），每 2 个 Q head 共享 1 对 KV，KV Cache 显存直接减半。GQA 不是独立优化，它的设计动机之一就是降低 KV Cache 的内存开销——在精度损失极小的前提下，把 cache 容量压下来。

    更进一步的优化方向（M2 范围外）： - MQA（Multi-Query Attention）：H_kv=1，所有 Q 共享一对 KV，cache 最小但精度损失更大 - PagedAttention（vLLM）：把 KV Cache 切成物理 page，按需分配，消除碎片 - 量化 KV Cache：把 K/V 存成 int8/fp8，进一步压缩显存

    2. 显存占用计算#

    2.1 计算公式#

    显存(bytes) = 2 × B × L × L_max × H_kv × D × dtype_bytes
              ↑K+V  ↑批   ↑层数  ↑最大长度  ↑KV头数 ↑头维度

    Qwen3-0.6B 参数代入（B=1，L_max=1024，bf16）：

    参数 符号 值 说明
    K+V 两份 — ×2 K 和 V 各一份
    batch size B ×1 单序列推理
    层数 L ×28 每层独立缓存
    最大序列长度 L_max ×1024 最大影响因子，线性
    KV head 数 H_kv ×8 GQA 已比 MHA 的 16 减半
    每头维度 D ×64 head_dim（注意不是 128）
    dtype — ×2 bytes bf16；fp32 翻倍 
    2 × 1 × 28 × 1024 × 8 × 64 × 2 = 58 MB

若 L_max=4096：  58 × 4 = 233 MB
若用 MHA（H_kv=16）：58 × 2 = 117 MB   ← GQA 的节省体现于此

    2.2 L_max（max_seq_len）如何选择#

    L_max 必须满足约束：T_p + N ≤ L_max（prompt 长度 + 生成长度 ≤ cache 容量）。

    做法 适合场景 缺点
    固定 L_max=1024 大多数对话场景，默认推荐 超长 prompt 报 IndexError
    L_max = T_p + max_new_tokens 精确，零浪费 每次 generate 需额外传参
    L_max = max_position_embeddings 模型支持的最大值 显存最大（Qwen3-0.6B 为 40960）

    M2 默认 L_max=1024，可通过 --max-seq-len CLI 参数覆盖。超出时触发 IndexError，明确报错优于静默截断。

    符号速查#

    本文其余章节用到的符号：

    符号 含义 Qwen3-0.6B 值
    T_p prompt 长度（prefill 阶段 token 数） 100～1024
    N 最大生成 token 数 100～512
    T 当前序列总长 = T_p + 已生成 token 数 ≤ L_max
    L_max max_seq_len，KV Cache 最大容量 1024（默认）
    L num_hidden_layers 28
    H_q num_attention_heads 16
    H_kv num_key_value_heads（GQA） 8
    D head_dim 64

    3. 代码架构与实现#

    inferlite M2 围绕"一次 prefill + N 次轻量 decode"共修改了 4 个核心文件。各文件的职责划分清晰：

    inferlite/
├── model/
│   ├── kv_cache.py     ← KV Cache 数据结构（新增）
│   ├── attention.py    ← Attention 扩展：支持 cache 读写（修改）
│   └── qwen3.py        ← Model.forward：透传 kv_cache 参数（修改）
└── engine/
    └── core.py         ← generate loop：prefill/decode 两阶段拆分（修改）

    M1 vs M2 核心差异一览#

    文件 M1 的做法 M2 的改变 为什么改
    core.py 每步 decode 把完整历史序列 input_ids 传入 model Prefill 一次处理 T_p 个 token；decode 每步只传当前 1 个 token M1 每步 forward 处理 T 个 token，计算量 O(T²)；M2 decode 只处理 1 个 token，O(T)
    core.py 单一 for 循环，每步结构相同 显式拆成 prefill 段 + decode 循环，cur_len 在 loop 里显式维护 两阶段的 position_ids 和 cache 写入位置不同，合并写容易产生隐蔽 bug
    attention.py hidden_states 每步传入完整 [B, T, H]，构建 T×T attn 矩阵 decode 时 hidden_states 是 [B, 1, H]；历史 K/V 从 cache 读，attn 矩阵变为 [1, T] 向量-矩阵乘 把 Attention 从 O(T²) 降至 O(T)
    attention.py 不存 K/V，每步重算 RoPE 后原地切片写入 cache，再切片读出历史 K/V 写入时序选 RoPE 之后（K 携带位置信息）、repeat_kv 之前（按 n_kv 维度存，省显存）
    kv_cache.py 无 新增 LayerKVCache（单层 K/V tensor 对）+ KVCache（28 层 + cur_len） 全局一次预分配，decode 阶段零动态 malloc

    3.1 KV Cache 数据结构#

    代码：inferlite/model/kv_cache.py

    设计思路：静态预分配，避免 decode 阶段动态 malloc；cur_len 作为唯一事实源，外置到 generate loop 显式维护，状态可见。

    @dataclass
class LayerKVCache:
    """单层 Transformer 的 KV Cache 容器。
    k/v shape 均为 [B, n_kv_heads, max_seq_len, head_dim]。
    有效范围由 KVCache.cur_len 决定，不依赖 tensor 内容。
    """
    k: torch.Tensor  # [B, n_kv_heads, max_seq_len, head_dim]
    v: torch.Tensor  # [B, n_kv_heads, max_seq_len, head_dim]


class KVCache:
    def __init__(self, layers: list[LayerKVCache]) -> None:
        self.layers = layers
        self.cur_len: int = 0  # 唯一事实源：当前已写入的有效 token 数

    @classmethod
    def from_config(cls, config, batch_size, max_seq_len, dtype, device) -> "KVCache":
        """一次性预分配所有层的 K/V tensor（零分配推理的前提）。"""
        layers = [
            LayerKVCache(
                k=torch.zeros(batch_size, config.num_key_value_heads,
                              max_seq_len, config.head_dim, dtype=dtype, device=device),
                v=torch.zeros(batch_size, config.num_key_value_heads,
                              max_seq_len, config.head_dim, dtype=dtype, device=device),
            )
            for _ in range(config.num_hidden_layers)
        ]
        return cls(layers)

    def reset(self) -> None:
        """重置 cur_len 为 0，不清零 tensor。
        下次 prefill 会从索引 0 覆盖写入，旧数据不影响正确性，
        避免对 58 MB tensor 做不必要的 fill_(0)。
        """
        self.cur_len = 0

    内存布局：

    预分配 tensor：shape [B=1, H_kv=8, L_max=1024, D=64]

位置索引：  0    1    2   ...  T_p-1  T_p  T_p+1 ...  L_max-1
           ┌─────────────────────┬──────────────────────────┐
prefill后：│K_0  K_1  K_2 ... K_Tp│  0   │  0   │  ...      │ cur_len=T_p
           └─────────────────────┘ ↑ 尾部零值，切片时不会读到

decode第1步后：
           │K_0  K_1 ... K_Tp  K_Tp+1│  0   │  0  │  ...   │ cur_len=T_p+1
                                      ↑ 追加写入

读取（decode 第 k 步）：
  full_k = k[:, :, :cur_len+1, :]   ← 切片只取有效范围

    3.2 Attention 读写逻辑#

    代码：inferlite/model/attention.py

    Attention 是 M1 → M2 变化最大的地方。M1 每步传入完整历史，attn 矩阵是 T×T；M2 decode 每步只传 1 个 token，K/V 从 cache 读，attn 矩阵退化为 1×T 向量乘：

    M1 decode step（无 cache）：
  输入：hidden [B, T, H]       ← 完整历史 T 个 token
  Q:    [B, n_q, T, D]
  K/V:  [B, n_q, T, D]         ← 每步从头重算
  QK^T: [B, n_q, T, T]         ← T×T 矩阵，O(T²)

M2 decode step（有 cache）：
  输入：hidden [B, 1, H]        ← 只有当前 1 个新 token
  Q:    [B, n_q, 1, D]
  K/V:  [B, n_q, T, D]          ← 从 cache 读取完整历史
  QK^T: [B, n_q, 1, T]          ← 1×T 向量-矩阵乘，O(T)

    接口扩展：GQAAttention.forward 新增两个可选参数，None 时退化为 M1：

    def forward(
    self,
    hidden_states: torch.Tensor,
    position_embeddings: tuple[torch.Tensor, torch.Tensor],  # (cos, sin)，由 Qwen3Model 统一计算
    layer_kv_cache: LayerKVCache | None = None,
    cache_position: int | None = None,  # = kv_cache.cur_len，由 generate loop 透传
) -> torch.Tensor:

    顺带优化：M1 中每层 Attention 各自调用 self.rotary_emb 计算 cos/sin（28 层 = 28 次重复）。M2 把 rotary_emb 移到 Qwen3Model.forward 统一计算一次，传入所有层，与 transformers 对齐。

    写入与读取（Prefill 和 Decode 共用同一段代码，T 决定行为）：

    if layer_kv_cache is not None:
    T = hidden_states.shape[1]  # Prefill: T=T_p；Decode: T=1

    # 写入当前 token(s)：切片赋值，原地写入，零内存分配
    # Prefill: cache_position=0，写入 [0, T_p)
    # Decode:  cache_position=cur_len，写入 [cur_len, cur_len+1)
    layer_kv_cache.k[:, :, cache_position:cache_position + T, :] = k
    layer_kv_cache.v[:, :, cache_position:cache_position + T, :] = v

    # 读取完整有效历史：切片只取有效范围，不含尾部零值
    k = layer_kv_cache.k[:, :, :cache_position + T, :]
    v = layer_kv_cache.v[:, :, :cache_position + T, :]
    写入方式 可读性 MPS 兼容 性能
    切片 [:, :, pos:pos+T, :]（inferlite 选择） ✅ 直接看到写哪里/读哪里 ✅ ✅ 零内存分配
    index_copy_（transformers） ❌ 需理解 API 语义 ⚠️ MPS 有兼容问题 ✅ 同样零分配
    torch.cat（DynamicCache） ❌ ✅ ❌ 每步申请新内存

    causal mask 用 T > 1 判断，而非 kv_cache is None：

    T   = hidden_states.shape[1]  # 当前输入长度
T_k = full_k.shape[2]         # K 的历史长度

if T > 1:  # prefill 需要因果 mask；decode T=1，没有"未来"，不需要
    causal_mask = torch.triu(
        torch.ones(T, T_k, dtype=torch.bool, device=device), diagonal=1
    )
    attn_scores = attn_scores.masked_fill(causal_mask[None, None], float("-inf"))

    causal mask 的需求来自"输入序列长度 > 1"，与有无 cache 无关：用 kv_cache is None 判断在 prefill 有 cache 时会导致 mask 丢失（silently wrong）。

    3.3 Generate Loop 拆分#

    代码：inferlite/engine/core.py

    M1 vs M2 在 generate loop 上的最核心差异：M1 每步把完整历史 token 传入 model；M2 把 prefill 和 decode 拆开，decode 每步只传1 个 token：

    # ─── M1：单一 for 循环，每步全量 forward ───────────────────────────────
#  decode 第 k 步的输入：[prompt + token_1 + ... + token_k]，长度 T_p+k
#  Attention 输入形状：hidden [B, T_p+k, H]，每步重建完整 T×T attn 矩阵
def generate_m1(engine, input_ids, max_new_tokens):
    for _ in range(max_new_tokens):
        logits = engine.model(input_ids)             # ← 每步传完整历史，T 越来越长
        next_token = sampler(logits[:, -1, :])
        input_ids = torch.cat([input_ids, next_token], dim=1)  # 历史序列不断增长
        if next_token.item() == eos_token_id:
            break
    return input_ids


# ─── M2：prefill + decode 显式拆分 ──────────────────────────────────
#  prefill：一次处理全部 prompt（必须，无法省略）
#  decode 第 k 步的输入：只有当前 1 个 token，历史 K/V 从 cache 读取
#  Attention 输入形状：hidden [B, 1, H]，attn 矩阵变为 [1, T] 向量-矩阵乘
def generate_m2(engine, input_ids, max_new_tokens, kv_cache):
    kv_cache.reset()

    # ── Prefill ──────────────────────────────────────────────────────
    T_p = input_ids.shape[1]
    position_ids = torch.arange(T_p).unsqueeze(0)        # [0, 1, ..., T_p-1]
    logits = engine.model(input_ids, position_ids, kv_cache)  # ← 一次写满 cache[0:T_p]
    kv_cache.cur_len = T_p

    next_token = sampler(logits[:, -1, :])
    generated = [next_token]

    # ── Decode：每步只传 1 个 token ──────────────────────────────────
    for _ in range(max_new_tokens - 1):
        position_ids = torch.tensor([[kv_cache.cur_len]])  # 绝对位置，不能写 [[0]]！
        logits = engine.model(next_token, position_ids, kv_cache)  # ← 每步只传 1 个 token
        kv_cache.cur_len += 1

        next_token = sampler(logits[:, -1, :])
        if next_token.item() == eos_token_id:
            break
        generated.append(next_token)

    return torch.cat([input_ids] + generated, dim=1)

    M1 的 generate() 每步都做 full forward。M2 拆成显式的 prefill + decode 两阶段，实际代码（向后兼容，kv_cache=None 退化为 M1）：

    def generate(engine, input_ids, max_new_tokens, kv_cache=None):
    if kv_cache is None:
        # M1 路径（原逻辑完全不变，向后兼容）
        for _ in range(max_new_tokens):
            next_token = engine.step(input_ids)
            input_ids = torch.cat([input_ids, next_token], dim=1)
            if next_token.item() == eos_token_id:
                break
        return input_ids

    # M2 路径 ────────────────────────────────────────
    kv_cache.reset()

    # Prefill：一次处理全部 prompt，写满 KV Cache 前 T_p 个位置
    T_p = input_ids.shape[1]
    position_ids = torch.arange(T_p).unsqueeze(0)        # [[0, 1, ..., T_p-1]]
    logits = engine.model(input_ids, position_ids, kv_cache)
    kv_cache.cur_len = T_p                               # 显式赋值，状态可见

    next_token = sampler(logits[:, -1, :])
    generated = [next_token]

    # Decode：每步只处理 1 个 token
    for _ in range(max_new_tokens - 1):
        position_ids = torch.tensor([[kv_cache.cur_len]])  # 绝对位置，不能写 [[0]]！
        logits = engine.model(next_token, position_ids, kv_cache)
        kv_cache.cur_len += 1                             # 显式递增

        next_token = sampler(logits[:, -1, :])
        if next_token.item() == eos_token_id:
            break
        generated.append(next_token)

    return torch.cat([input_ids] + generated, dim=1)

    position_ids 必须用绝对位置（[[cur_len]]），不能写 [[0]]：

    decode step t：
  正确：position_ids = [[T_p + t]]  →  q 与历史 k_n 的相对距离 = (T_p+t) - n  ✅
  错误：position_ids = [[0]]        →  q 与历史 k_n 的相对距离 = 0 - n        ❌

RoPE 正确性依赖相对距离编码。写 [[0]] 时推理不报错，但位置编码全部错误，
生成质量严重下降——这是最危险的沉默 bug 类型。

    完整数据流（以 decode 步为例）：

    generate()  [core.py]
  │  position_ids = [[cur_len]]
  │  engine.model(cur_token, position_ids, kv_cache)
  │
  ↓
Qwen3Model.forward()  [qwen3.py]
  │  position_embeddings = rotary_emb(hidden, position_ids)  ← 统一计算一次
  │
  └─ for i in range(L=28):
       DecoderLayer[i](hidden, position_embeddings,
                       layer_kv_cache=kv_cache.layers[i],
                       cache_position=kv_cache.cur_len)
         └─ GQAAttention.forward()  [attention.py]
              ├─ q/k/v projection
              ├─ q_norm / k_norm / RoPE
              ├─ 写入：kv.k[:,:,cur_len:cur_len+1,:] = k
              ├─ 读取：full_k = kv.k[:,:,:cur_len+1,:]
              └─ Attention(q, full_k, full_v) → output

    4. 与主流框架对比#

    维度 transformers StaticCache nano-vllm/vLLM inferlite M2
    内存策略 预分配连续 tensor，max_len 固定 全局预分配，block 粒度动态分配（PagedAttention） 预分配连续 tensor，max_len 固定
    写入方式 index_copy_ 原地写 Triton kernel，slot 散列写 切片写入 [:,:,pos:pos+T,:]
    读取方式 返回完整 tensor（含尾部零值，需 mask） FlashAttn + block_table 间接读 切片读有效范围 [:,:,:pos+T,:]
    cur_len 管理 藏在 cache 对象（cumulative_length） 调度器管理，外部不感知 generate loop 显式维护
    position_ids Model.forward 自动推算（隐式） 调度器管理（隐式） generate loop 显式传入
    多序列 需 padding mask 天然支持 continuous batching 不支持（M3 范围）
    torch.compile ✅（StaticCache 专为此设计） N/A（Triton kernel） ❌（切片边界动态）
    MPS 支持 ⚠️（index_copy_ 有 fallback） ❌（需 CUDA） ✅
    依赖 纯 PyTorch Triton + FlashAttention 纯 PyTorch

    inferlite M2 的定位：与 transformers StaticCache 最接近，但"切片写入替代 index_copy_"（MPS 兼容 + 更可读）、"cur_len 外置显式维护"（状态可见，教学目的）。

    5. 实测性能#

    5.1 测试方法#

    # 脚本：scripts/bench_kv_cache.py
# 扫描不同 prompt 长度，对比 M1/M2 tok/s
uv run python scripts/bench_kv_cache.py \
  --model-dir ~/.cache/modelscope/hub/models/Qwen/Qwen3-0___6B \
  --device mps --dtype bf16 --gen-tokens 128

    方法：用 pad_token 构造固定长度合成 prompt（控制变量），每种长度跑一次，time.perf_counter 计时，warmup 1 次（让 MPS 完成 JIT 编译）。

    5.2 结果（Mac M3 Pro，MPS，bfloat16，gen_tokens=128）#

    prompt_tokens M1 tok/s M2 tok/s Speedup
    32 13.8 24.9 1.80×
    64 12.7 28.9 2.27×
    128 9.6 25.7 2.67×
    256 6.2 24.1 3.91×
    512 3.3 24.1 7.36×

    规律： - M2 tok/s 基本稳定在 24-29（每步固定处理 1 个 token，O(T) 恒定） - M1 tok/s 随 prompt 长度线性下降（O(T²) 逐渐显现） - Speedup 随 prompt 长度线性增长，符合 O(T²)/O(T) = T 倍的理论预期

    5.3 理论 ~600× vs 实测 7.36× 的差距#

    理论加速比假设 Attention 是唯一瓶颈。实测中，每步 decode 还需把 28 层权重（~1.2 GB）从内存读一遍：

    单步 decode 总耗时 ≈ 权重 I/O 时间  +  Attention 计算时间
                      ↑ M1/M2 共有        ↑ M2 省去了这部分

Qwen3-0.6B on MPS：
  权重 I/O：1.2 GB ÷ 200 GB/s ≈ 6 ms/step（固定）
  Attention（T=512，无 cache）：~5 ms
  → 总耗时 ≈ 11 ms；M2 总耗时 ≈ 6 ms → 端到端加速比 ≈ 1.8×（而非理论 512×）

    解耦验证：单独测量 Attention 层（不含任何权重 I/O），可以看到 KV Cache 对 Attention 本身的真实加速：

    脚本：scripts/bench_attention_only.py

    Attention-only benchmark（Mac M3 Pro, MPS, bfloat16, reps=20）

seq_len   M1 (ms)   M2 (ms)   Speedup   理论值
-----------------------------------------------
     32      0.81      0.69     1.2×      32×
     64      1.20      0.49     2.4×      64×
    128      1.43      0.56     2.5×     128×
    256      2.50      0.49     5.1×     256×
    512      5.58      0.68     8.2×     512×
   1024     14.72      1.00    14.7×    1024×

    M1 耗时随 T 增长（0.81 ms → 14.72 ms），趋势接近 T²。M2 耗时基本稳定在 0.5–1 ms（MPS kernel launch 固定开销）。实测 Speedup 仍低于理论 T 倍，原因是 MPS 单次 kernel 有约 0.5 ms 的 launch overhead，短序列时两者都受这个下限约束，序列更长后差距会趋近理论值。

    两组数据对比如下：

    T 端到端 Speedup（含权重 I/O） Attention-only Speedup
    32 1.8× 1.2×
    256 3.9× 5.1×
    512 7.4× 8.2×
    1024 ~14×（预估） 14.7×

    T=512 时 Attention-only 加速已达 8.2×，与端到端 7.4× 接近——这说明在 T=512 时，Attention 计算时间已经与权重 I/O 相当，KV Cache 的收益开始"透出来"。更长上下文（T>1024）时 Attention 会彻底主导，端到端加速比也会继续拉近理论值。

    Attention-only Speedup 仍远低于理论 T 倍的原因#

    即便剥离了权重 I/O，T=1024 时实测 14.7× 仍远低于理论 1024×，有三层原因：

    ① kernel launch 固定开销（约 0.54 ms）

    一次 GQAAttention.forward 包含约 15–20 个 PyTorch 操作（q/k/v proj、RMSNorm、RoPE、QK^T、softmax、AV、o_proj……），每个操作在 MPS 上都是一次独立的 kernel launch，固定开销约 20–50 µs/kernel：

    M2 固定开销 ≈ 18 个 kernel × 30 µs ≈ 0.54 ms（与 T 无关）

T=1024 时 M2=1.00 ms，其中固定开销已占 54%
→ 分母被抬高，直接压低 Speedup

    ② M2 的 Attention 不是 O(1)，是 O(T)

    M2 decode 步做的是 [1, D] × [T, D]^T（1 个 query 对 T 个历史 key 的点积），计算量随 T 线性增长，不是常数：

    M1 Attention 计算量 ∝ T²（T 个 query，每个对 T 个 key）
M2 Attention 计算量 ∝ T（1 个 query，对 T 个 key）

从数据反推 M2 的 O(T) 项：
  T=64:   M2=0.49 ms，固定开销主导，线性项 ≈ 0
  T=512:  M2=0.68 ms，线性项 ≈ 0.14 ms
  T=1024: M2=1.00 ms，线性项 ≈ 0.46 ms

    ③ M1 的 proj 在大 T 时 arithmetic intensity 更高，效率反而更好

    M1 每步处理 T=1024 个 token，proj 的矩阵乘法是 [1024, H] × [H, H]；M2 只处理 1 个 token，是 [1, H] × [H, H]（向量-矩阵乘）。前者 arithmetic intensity（FLOPs/Byte）高于后者，更接近 GPU roofline 的计算瓶颈区，执行效率更高。因此 M1 实际比"理论 T² 倍慢"的预期快，使分子没那么大，Speedup 被进一步压缩。

    要突破这个瓶颈需要算子融合：FlashAttention 把 QK^T + softmax + AV 合并为一个 kernel（消除中间 kernel launch），并在 SRAM 内完成计算（减少显存带宽），可以把 M2 固定开销压到 µs 级，让 Speedup 更接近理论 T 倍。

    KV Cache 的核心价值在： 1. 长 prompt（prompt 越长，Attention 在总耗时中占比越高，加速比越大） 2. M3 Continuous Batching（多请求共享一次权重读取，吞吐量才真正拉开差距） 3. 配合 FlashAttention（消除 kernel launch overhead，让 Attention-only 加速比贴近理论值）

    6. 踩坑与教训#

    坑 现象 根因 修复
    position_ids = [[0]] 推理不报错，但生成质量差、输出混乱 decode 步 RoPE 位置编码全在位置 0，相对距离信息错误 [[cur_len]] 绝对位置
    causal mask 用 if kv_cache is None 判断 prefill 有 cache 时 mask 丢失，logits 错误 prefill/decode 的区分是 T>1，与有无 cache 无关 改为 if T > 1
    reset() 是否要清零 tensor 不清零也完全正确 prefill 覆盖写入 [0, T_p)，旧数据不影响 只清 cur_len，无需 fill_(0)
    AutoTokenizer.from_pretrained 报 HFValidationError 本地路径被当作 HF repo id 验证，报错 transformers 5.x 对本地路径的校验更严格 加 local_files_only=True
    ModelScope 缓存路径 . 变 ___ 手动拼路径时找不到文件 ModelScope 对文件系统特殊字符的安全替换（0.6B → 0___6B） 用 find 确认实际路径

    8. 下一步：M3 Continuous Batching#

    M2 留下三个接口桩，M3 在此基础上生长：

    1. KVCache 是单序列设计

    cur_len 只有一个，意味着 batch 内所有序列同步推进。M3 需要每个请求有独立的 cur_len，或改为 PagedAttention 的 block_table 管理（逻辑地址→物理 block 映射）。

    2. generate() 是单请求循环

    M3 的 FCFSScheduler 把多个请求合并进同一个 forward，共享一次权重读取。这是吞吐量的真正来源：

    单序列 KV Cache：省去的是 Attention 的 O(T²) 计算
Continuous Batching：省去的是每请求独立读一遍权重的 I/O

    两者叠加才能接近生产级 LLM 推理服务的吞吐量。

    3. device/dtype 已就绪

    M2 T5 完成的 --device mps/cuda + bf16 支持是 M3 的先决条件，可直接复用。
